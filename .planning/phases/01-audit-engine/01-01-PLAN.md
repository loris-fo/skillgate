    - schema.test.ts: Invalid score enum value (e.g. "unknown") fails validation
    - schema.test.ts: All 5 category names are required (missing one fails)
    - schema.test.ts: by_design boolean required per category
    - schema.test.ts: utility_analysis fields validated (what_it_does string, use_cases array, etc.)
    - schema.test.ts: recommendation verdict enum enforced (only install/install_with_caution/review_first/avoid)
    - hash.test.ts: Same content produces same hash
    - hash.test.ts: Different content produces different hash
    - hash.test.ts: Content differing only in whitespace/line endings produces same hash (normalization works)
    - hash.test.ts: BOM character stripped before hashing
    - hash.test.ts: Cache key includes prompt version (format: {hash}:{version})
    - hash.test.ts: Tab characters collapsed to single space
  </behavior>
  <action>
1. **types.ts**: Define TypeScript types (these are the source of truth, Zod schema infers from them or mirrors them):
   - `Score = "safe" | "low" | "moderate" | "high" | "critical"`
   - `Verdict = "install" | "install_with_caution" | "review_first" | "avoid"`
   - `CategoryResult = { score: Score; finding: string; detail: string; by_design: boolean }`
   - `Categories = { hidden_logic, data_access, action_risk, permission_scope, override_attempts }` (each CategoryResult)
   - `UtilityAnalysis = { what_it_does: string; use_cases: string[]; not_for: string[]; trigger_behavior: string; dependencies: string[] }`
   - `Recommendation = { verdict: Verdict; for_who: string; caveats: string[]; alternatives: string[] }`
   - `AuditResult = { overall_score: Score; verdict: string; summary: string; intent: string; categories: Categories; utility_analysis: UtilityAnalysis; recommendation: Recommendation }`
   - `AuditError` class extending Error with a `code` field (INPUT_TOO_LARGE, API_ERROR, VALIDATION_ERROR)

2. **schema.ts**:
   - Zod schema exactly matching AuditResult type (use RESEARCH.md Zod example as base). Use `z.infer<typeof auditResultSchema>` to verify type alignment.
   - Anthropic tool definition `AUDIT_TOOL` with `name: "record_audit"`. IMPORTANT: Do NOT use $ref in the JSON schema -- inline the category object schema for each of the 5 categories (per Pitfall 1 in RESEARCH.md). The tool input_schema must include all fields: overall_score, verdict, summary, intent, categories (with all 5 inlined), utility_analysis, recommendation.
   - Export both `auditResultSchema` and `AUDIT_TOOL`.

3. **hash.ts**:
   - `PROMPT_VERSION = "1.0"` constant.
   - `normalizeContent(content: string): string` -- trim, strip BOM (\uFEFF), normalize line endings (\r\n and \r to \n), collapse horizontal whitespace (spaces and tabs to single space), collapse 3+ consecutive newlines to 2, apply Unicode NFC normalization.
   - `buildCacheKey(content: string): string` -- normalize content, SHA-256 hash (hex), return `{hash}:{PROMPT_VERSION}`.
   - Export PROMPT_VERSION, normalizeContent, buildCacheKey.

4. **tests/schema.test.ts**: Write tests per behavior list above. Create a valid `mockAuditResult` fixture object for reuse.

5. **tests/hash.test.ts**: Write tests per behavior list above.

Run tests with `pnpm --filter @skillgate/audit-engine test` after implementation.
